# 多端产品设计规范

## 新项目启动门禁

面向带界面的项目，AI 开工前必须先判断当前项目是新项目、半成品项目，还是已有产品迭代。判断不清时按新项目处理，先沟通再动手；别上来就写组件，写完发现审美方向南辕北辙，那就成了电子返工。

### 新项目判定
- **用户明确表达**：用户说“新项目”“从零开始”“做一个系统/工具/App/网站”“搭一个 MVP”等，视为新项目
- **仓库结构很空**：缺少 `README.md`、`PRODUCT_OVERVIEW.md`、`AGENTS.md`、`CLAUDE.md`、主要源码目录或只有脚手架文件，视为新项目
- **没有产品定义**：找不到目标用户、核心流程、信息架构、设计系统、版本策略和运行方式说明，视为新项目
- **没有 UI 体系**：找不到颜色、字体、字号、间距、圆角、组件状态、图标、布局断点等 token 约定，界面类项目视为设计未启动
- **已有项目信号**：存在稳定源码、线上版本、CHANGELOG、产品概览、设计系统、既有页面风格和测试流程时，按迭代项目处理，优先继承现有约定
- **模糊情况**：只要存在 30% 以上不确定性，先向用户确认，不要假装自己会读心术，鲁迅先生没说过这话但大意可以有

### 开工流程
1. **扫描项目现状**：先查看 `README.md`、`PRODUCT_OVERVIEW.md`、`AGENTS.md`、`CLAUDE.md`、设计规范、源码目录、构建配置和版本文件
2. **判断项目阶段**：明确反馈“新项目 / 半成品 / 既有项目迭代”，并说明判断依据
3. **提出设计问题**：新项目或设计不完整时，先询问平台、用户、场景、视觉气质、信息密度、设计参照、UI 体系偏好、多语言计划和首版范围
4. **生成启动文档**：先生成或更新 `AGENTS.md`、`CLAUDE.md`、`PRODUCT_OVERVIEW.md` 或项目约定的等价文档，再进入正式实现
5. **确认默认假设**：如果用户要求直接开工，必须先写明默认设计假设，并把假设落入启动文档
6. **实现时受文档约束**：后续页面、组件、样式、文案、版本和测试必须遵守启动文档；需求变化时同步更新文档

### 新项目第一轮确认清单

第一轮沟通只确认会改变产品方向、设计系统或工程架构的事项。不要把下面每个子项都机械拆成问题；优先合并成 6-8 个高价值问题，能从仓库、截图、现有文档判断出来的内容直接写成默认假设让用户确认。

| 主题 | 需要确认到位 | 为什么要问 |
|------|--------------|------------|
| 项目定位 | 项目一句话、服务对象、解决的问题、首版成功标准 | 防止做成“看起来都对，但不是用户要的东西” |
| 用户与场景 | 主要角色、前三个高频任务、关键路径、最不能出错的流程 | 决定导航、信息架构、权限和验收主流程 |
| 首版范围 | 必须有、可以后置、明确不做、时间/预算约束 | 控制 MVP 边界，避免一开局就摊大饼 |
| 平台与设备 | H5、电脑版网页、macOS、Android、后台、大屏、目标尺寸和兼容范围 | 决定布局、交互方式、响应式和测试矩阵 |
| 设计方向 | 视觉气质、信息密度、品牌色、竞品/参考、组件库、UI token 精细度 | 决定字体、字号、颜色、间距、圆角、图标和组件状态 |
| 多语言与内容 | 首发语言、后续语言、默认/fallback、术语表、日期/数字/货币格式、RTL | 影响 i18n 架构、文案长度、字体 fallback 和布局预留 |
| 数据库、数据与权限 | 数据库选型（SQLite/MySQL/PostgreSQL 等）、开发/生产是否一致、迁移工具、数据来源、角色权限、敏感字段、导入导出、全文检索、备份恢复、第三方集成、审计需求 | 影响数据模型、事务、查询、部署、迁移、权限、安全和回滚 |
| 交付与验收 | 技术栈偏好、包管理、部署方式、版本号、必须跑的测试、性能/无障碍目标 | 让“完成”有标准，别靠一句“差不多了”收工 |

### 沟通节奏
- **先声明判断**：先说“我判断这是新项目 / 半成品 / 迭代项目”，再列依据
- **第一轮最多 8 个主题**：每个主题尽量问一个组合问题，避免把用户问成填表机器
- **先给推荐方案**：不要把架构、数据库、技术栈、UI 体系全丢给用户选择；AI 必须先给最佳实践建议，再让用户确认或指出例外
- **带默认选项**：如果用户没有偏好，给出推荐默认值，例如“React + TypeScript + yarn + i18n 字典 + PostgreSQL + 标准密度”
- **先问不可逆事项**：平台、用户、范围、设计方向、数据库选型、权限安全、多语言计划优先；按钮文案这种可后改的别抢第一轮
- **边问边沉淀**：用户回答后立即写入 `AGENTS.md` / `CLAUDE.md` / `PRODUCT_OVERVIEW.md`，不要只停留在聊天记录里
- **允许渐进确认**：用户想快速开工时，先落默认假设并标注“待确认”，实现到相关模块前再补问

### 推荐优先，确认兜底

新项目沟通不要只会问“你想用什么”。AI 要先根据项目目标、部署环境、数据量、协作人数、平台和现有仓库信号给出推荐方案，然后请用户确认。

| 领域 | 默认最佳实践 | 适用情况 | 需要用户确认的例外 |
|------|--------------|----------|--------------------|
| 应用架构 | 先做模块化单体，清晰分层，保留接口边界 | 绝大多数新项目、MVP、内部系统、早期 SaaS | 已有微服务体系、强隔离团队、独立扩缩容或合规边界 |
| 前端技术栈 | React + TypeScript + yarn + Vite/Next.js，文案走 i18n 字典 | 新 Web/H5/后台项目 | 老项目已有 Vue/Angular/其他栈，或 SEO/SSR 有特殊要求 |
| 后端接口 | REST/JSON 起步，按资源建模，错误格式统一，OpenAPI 可选 | 常规业务系统和后台 | 强实时、复杂订阅、跨团队契约需 GraphQL/gRPC/Event |
| 数据库 | 多用户服务端默认 PostgreSQL；已有 MySQL 体系则沿用 MySQL；单机/桌面/轻量原型可用 SQLite | 大多数新全栈项目 | 运维团队只支持某种数据库、需要嵌入式离线、强 JSON/全文/地理能力、云厂商限制 |
| 开发/生产数据库 | 尽量同一数据库引擎；本地用 Docker 或托管 dev 实例 | 避免 SQLite 开发、PostgreSQL 生产导致 SQL/迁移差异 | 桌面 App、本地优先产品、纯原型 |
| 数据迁移 | 一开始就引入迁移工具，迁移文件可回滚，随版本记录 | 任何有持久化的项目 | 临时 demo 且明确不会保留数据 |
| 权限模型 | 先定义用户、角色、权限边界；后台默认 RBAC 起步 | 多角色后台、SaaS、内部工具 | 纯个人工具或无登录公开页面 |
| 部署形态 | MVP 优先简单可回滚：单服务/单二进制/Docker 单体，健康检查和版本可观测 | 小团队、早期产品、内部系统 | 已有 K8s/多区域/高可用基础设施 |
| UI 体系 | 先落最小 token：字体、字号、颜色、间距、圆角、组件状态 | 所有界面项目 | 已有设计系统或品牌规范 |
| 测试策略 | 核心逻辑单测 + 关键流程 E2E + 构建检查 | 大多数项目 | 高合规、高资金风险、医疗金融等需更高覆盖 |

### UI 规范方向推荐

涉及界面项目时，AI 不要只问“你喜欢什么风格”。先根据平台和应用类型给出 2-4 套适合的 UI 规范方向，让用户选择或排除；用户决策几轮后，再细化成 `docs/design-system.md`、`docs/ui-tokens.md` 或项目约定的等价文档。

| 平台 / 类型 | 推荐方向 | 适用场景 | 默认建议 |
|-------------|----------|----------|----------|
| Web 后台 / SaaS / CRM | 稳重工作台、现代科技、数据分析、企业品牌 | 长时间使用、表格表单多、需要高效率扫描 | 默认稳重工作台：中性配色、标准/紧凑密度、清晰表格、少装饰 |
| Web 官网 / 营销页 | 品牌展示、现代科技、内容编辑、活动视觉 | 品牌表达、转化、介绍产品或服务 | 根据品牌资产选择；没有品牌时用现代科技但控制渐变和装饰 |
| H5 / 移动 Web | 轻量工具、流程表单、活动视觉、内容阅读 | WebView、移动浏览器、分享传播、短流程 | 默认轻量工具：单列、底部主操作、弱网反馈、触控优先 |
| iOS App | Apple 原生克制、内容消费、游戏/卡通、高级生产力 | iPhone/iPad 原生体验 | 默认 Apple 原生克制；游戏/儿童/娱乐产品才优先卡通化 |
| macOS App | 原生生产力、专业创作、开发者工具、菜单栏轻工具 | 桌面效率工具、专业软件、长期使用 | 默认原生生产力：侧边栏、工具栏、菜单栏、快捷键体系 |
| Android App | Material 3 生产力、内容消费、游戏/卡通、本地工具 | Android 原生体验、平板/折叠屏 | 默认 Material 3 生产力；消费类可增强动效和内容沉浸 |
| 游戏 / 娱乐 | 卡通、像素、写实、科幻、暗黑、童趣 | 游戏、互动体验、强视觉记忆 | 先确认受众年龄、题材和资产生产方式，再定风格 |
| 数据大屏 / BI | 高对比数据屏、运营驾驶舱、浅色分析台 | 指标监控、投屏、管理驾驶舱 | 默认浅色分析台；真大屏再用高对比深色方案 |

选择后必须继续细化：
- **视觉方向**：关键词、参考产品、禁用风格、品牌感强弱
- **Token**：字体、字号、颜色、间距、圆角、阴影、断点、动效时长
- **组件规范**：按钮、输入、表格、列表、弹窗、抽屉、Toast、空状态、错误态
- **平台差异**：Web 的 hover/键盘、移动端触控、macOS 菜单栏/快捷键、Android 返回手势
- **验收样例**：至少定义 2-3 个代表页面或流程，用来检查规范是否落地

确认方式建议：

```markdown
基于这个项目，我建议先从这几套 UI 方向里选：
1. 稳重工作台：适合后台/SaaS，信息密度高，少装饰
2. 现代科技：适合 AI/开发者/数据产品，视觉更轻快
3. 内容品牌：适合官网/展示页，强调叙事和品牌记忆

我建议默认选【稳重工作台】，因为你的核心场景是长时间操作和数据扫描。你可以确认、换方向，或给一个参考截图。
```

### 文档优先，不靠 Memory

架构、数据库、设计规范、UI token、多语言计划、部署方式、验收标准等关键决策必须落到仓库文档里。Memory 可以作为提醒，但不能作为唯一事实来源；跨 Agent、跨线程、从 Codex 换到 Claude 时，只有仓库文档最可靠。

新项目建议至少维护：
- **`AGENTS.md` / `CLAUDE.md`**：AI 协作规则入口，写明当前项目阶段、关键约束和必须遵守的文档
- **`docs/product-brief.md`**：产品定位、用户、场景、首版范围、暂不做
- **`docs/architecture.md`**：技术栈、模块边界、数据流、接口风格、部署形态
- **`docs/design-system.md`**：UI 方向、设计原则、组件规范、状态规范、平台差异
- **`docs/ui-tokens.md`**：颜色、字体、字号、间距、圆角、阴影、断点、动效
- **`docs/i18n.md`**：语言计划、默认/fallback、术语表、日期/数字/货币格式、RTL
- **`docs/decisions/`**：重要架构和设计决策记录，推荐用 ADR 格式

规则：
- **先写文档再大规模实现**：新项目方向、架构和 UI 体系未落文档前，不进入大规模编码
- **变更同步文档**：如果实现过程中改了数据库、框架、设计方向、组件规范或多语言策略，必须同步更新对应文档
- **文档可被工具生成**：脚本可以生成初版 `CLAUDE.md`，但项目内最终约束应落在仓库文档里
- **聊天记录不算文档**：聊天里确认过但未写入仓库的约定，后续 Agent 不应默认知道

确认方式建议：

```markdown
基于当前项目，我建议先采用：
- 架构：模块化单体，避免过早微服务
- 前端：React + TypeScript + yarn
- 数据库：PostgreSQL（如果只是本地桌面/单机工具则改 SQLite）
- 迁移：从第一版开始使用迁移工具
- 部署：先做简单可回滚部署，暴露版本号和健康检查

这些是默认建议。你只需要告诉我哪些不符合你的实际约束；没异议我就写进启动文档并按这个执行。
```

### 推荐提问模板
```markdown
我先判断这是一个【新项目/半成品/既有项目迭代】，依据是：...

为了避免方向跑偏，我先给一套默认建议，再确认 8 件事；你可以简单回答，不确定的我会按推荐方案写入启动文档：

1. 项目一句话和目标用户是什么？
2. 首版最重要的 3 个任务是什么，哪些明确不做？
3. 目标平台和主要设备尺寸是什么？
4. 你希望视觉气质偏工具型、品牌型、内容型、数据型，还是其他？有没有参考产品或截图？
5. UI 体系是否有现成规范？如果没有，我是否按默认字体、字号、颜色、间距和组件状态先建一套？
6. 首发和后续计划支持哪些语言？默认语言和 fallback 语言是什么？
7. 我建议多用户服务端默认 PostgreSQL、单机/桌面/轻量原型用 SQLite、已有 MySQL 运维体系则用 MySQL；你的项目是否有不同约束？
8. 登录权限、敏感信息、第三方集成、技术栈、部署方式、测试命令和验收标准有什么偏好？
```

### 启动文档必须包含
- **项目定位**：一句话说明项目做什么、给谁用、解决什么问题
- **平台范围**：H5、电脑版网页、macOS App、Android App、服务端、后台管理等
- **目标用户与核心任务**：主要角色、前三个高频任务、关键路径和不可做错的流程
- **设计原则**：视觉气质、信息密度、导航模式、首屏策略、空/错/加载状态原则
- **UI 体系**：字体族、字号体系、字重、字间距、颜色 token、间距、圆角、阴影、图标、组件状态和动效
- **多语言计划**：首发语言、后续支持语言、默认语言、fallback 语言、日期/时间/数字/货币格式、RTL 需求
- **技术栈约束**：框架、包管理器、i18n、状态管理、样式方案、测试工具、构建和部署方式
- **数据库与数据策略**：SQLite/MySQL/PostgreSQL 等选型、开发/生产差异、迁移工具、数据来源、备份恢复、全文检索、导入导出和事务要求
- **版本与可观测性**：语义化版本、CHANGELOG、版本展示、服务端版本 header、日志和埋点要求
- **安全与隐私**：密钥管理、敏感信息脱敏、权限、输入校验、依赖审计
- **验收标准**：主流程、响应式范围、无障碍、性能预算、测试命令和已知风险

## 新项目启动前的设计沟通

面向 H5、电脑版网页、macOS App、Android App 等带界面的新项目，AI 在动手设计或编码前必须先和用户确认设计规范。不要一上来就堆页面，先把方向问清楚；界面不是煎饼果子，不能什么都往里加。

### 必问问题
- **目标平台**：本次优先做 H5、电脑版网页、macOS App、Android App，还是多端统一设计
- **核心用户**：给谁用，是内部运营、开发者、普通消费者、管理者，还是特定行业角色
- **核心场景**：用户打开后的前三个高频任务是什么，是否有必须优先完成的关键流程
- **视觉气质**：偏工具型、品牌展示型、内容阅读型、数据分析型、创作型，还是交易/运营型
- **设计参照**：是否有已有品牌色、设计系统、竞品、截图、Figma、Apple/Material/Web 规范约束
- **信息密度**：偏紧凑高效，还是留白舒展；是否需要适配大屏、横屏、折叠屏或小屏
- **国际化与文案**：首发支持哪些语言、后续计划支持哪些语言、默认语言是什么、fallback 语言是什么、是否需要 RTL、数字/日期/货币格式、是否需要无障碍读屏文案
- **上线约束**：首版范围、设备兼容、性能目标、审核要求、版本展示与埋点要求

### 输出约定
- **先给设计简报**：总结平台、用户、任务、风格、导航、关键组件、响应式策略和风险点
- **再做界面方案**：用户确认或补充后，再进入原型、页面实现或代码修改
- **多端先定主端**：多端项目先确认主端体验，再决定复用、裁剪或重排策略
- **无法确认时给默认假设**：如果用户要求直接开工，必须显式列出采用的默认设计规范
- **需求变化要回写**：项目推进中如果平台、用户或核心流程变化，应同步更新设计简报或 `PRODUCT_OVERVIEW.md`

## UI 体系最小集

新项目只要涉及界面，就必须先定义一套最小 UI 体系。可以简洁，但不能缺席。

### 字体与字号
- **全局字体族**：Web 项目优先采用 `Inter, -apple-system, BlinkMacSystemFont, "PingFang SC", "Microsoft YaHei", sans-serif`；如品牌或平台已有规范，以既有规范为准
- **等宽字体族**：代码、密钥、ID、模型名、日志等使用 `ui-monospace, SFMono-Regular, Menlo, Monaco, Consolas, monospace`
- **多语言字体回退**：必须按计划支持语言定义字体 fallback；中文界面显式考虑 `PingFang SC`、`Microsoft YaHei`，日文/韩文/泰文/阿拉伯文等语言要确认系统字体和行高策略
- **字号体系**：至少定义页面标题、分区标题、正文、辅助说明、按钮、表格、指标数字、代码文本
- **字重与字间距**：标题、指标数字、品牌名等需要明确字重；如使用紧凑后台风格，字间距要谨慎设定并验证可读性

### 多语言与本地化
- **先问语言范围**：新项目必须询问首发支持语言和未来计划支持语言，不能默认只有中文或英文
- **默认与回退**：必须明确默认语言、fallback 语言和缺失翻译时的展示策略
- **格式本地化**：日期、时间、数字、货币、百分比、姓名、地址、电话号码等格式必须按目标市场处理
- **版面预留**：英文、德文、俄文等文案可能明显变长，中文、日文、韩文换行规则不同，按钮、表格列、导航和卡片必须预留伸缩空间
- **RTL 预判**：如果计划支持阿拉伯语、希伯来语等 RTL 语言，布局、图标方向、输入框和动画方向要在设计阶段确认
- **文案来源**：文案统一进入 i18n / 本地化资源，禁止把多语言字符串散落在组件里

示例：

| Token | 字号 | 字重 | 字间距 | 用途 |
|------|------|------|--------|------|
| `metric-value` | 22px | 700 | -0.5px | 指标卡片 KPI 数值 |
| `page-title` | 20px | 700 | -0.3px | 页面主标题 |
| `section-title` | 16px | 700 | -0.2px | 通用分区标题 |
| `body` | 14px | 400 | 0 | 正文和表格内容 |
| `caption` | 12px | 400 | 0 | 辅助说明 |
| `code` | 13px | 500 | 0 | 代码、ID、密钥、日志 |

### 颜色与语义
- **颜色 token 化**：至少定义 `background`、`surface`、`text-primary`、`text-secondary`、`border`、`primary`、`success`、`warning`、`danger`
- **状态色固定语义**：不要让红色同时表示品牌、危险和涨幅；一个颜色多头兼职，迟早出事
- **深浅色策略**：桌面 Web、macOS、Android 项目要明确是否支持深色模式
- **对比度验证**：正文、按钮、错误提示、禁用态都要满足可读性要求

### 间距、圆角与布局
- **间距 scale**：至少定义 4、8、12、16、24、32px 或等价 scale
- **圆角规则**：后台和工具型界面默认克制，卡片、按钮、输入框、弹窗各自明确半径
- **布局断点**：Web/H5 至少定义移动、平板、桌面断点；App 至少定义小屏、大屏、横屏/竖屏策略
- **密度等级**：区分舒展、标准、紧凑三种密度，后台/数据产品默认标准或紧凑

### 组件状态
- **基础组件**：按钮、输入框、选择器、表格/列表、卡片、标签、弹窗、抽屉、Toast、Tooltip、空状态
- **状态覆盖**：default、hover、active、focus、disabled、loading、selected、error 必须有视觉定义
- **危险操作**：删除、覆盖、停用、发布、付款等必须有确认、权限和可恢复策略
- **文案来源**：所有可见文案必须走 i18n 或平台本地化资源，不能散落在视图代码里

## 通用界面原则
- **任务优先于装饰**：界面首先服务用户完成任务，视觉风格不能抢走主要操作的注意力
- **首屏展示真实产品状态**：应用、工具、游戏、数据产品首屏应直接进入可用体验，不做空泛落地页
- **导航稳定可预期**：主导航、返回、保存、取消、危险操作等行为要符合平台习惯
- **状态完整**：加载、空状态、错误、无权限、离线、成功反馈、撤销和确认都要设计
- **组件语义清晰**：按钮用于命令，开关用于二元状态，分段控件用于模式，菜单用于选项集合
- **触控与键鼠分开考虑**：移动端不依赖 hover，桌面端要支持 hover、右键、快捷键和焦点态
- **文案走本地化资源**：React、Android、macOS 等项目的可见文案不得硬编码在视图代码里
- **无障碍不是加分项**：颜色对比、键盘焦点、读屏标签、字号缩放、错误提示都要可用
- **性能影响设计**：图片、动画、图表、列表和首屏资源要有预算，别让用户看着加载条修身养性
- **版本可见**：工具型产品、后台、App 设置页或关于页应展示当前版本号，便于排查问题

## H5 网页

适用于手机浏览器、微信/飞书等内嵌 WebView、活动页、轻量工具页和移动端 Web 应用。

### 布局
- **移动优先**：优先覆盖 360px、375px、390px、414px、430px 宽度，再向平板扩展
- **安全区适配**：底部操作栏、浮动按钮、Toast、弹窗要考虑刘海屏和系统手势区域
- **单列为主**：核心流程用单列纵向推进，复杂信息拆成步骤、折叠区或分段标签
- **关键操作贴近拇指区**：主要按钮放在底部固定区或内容流末端，避免让用户在小屏上来回够
- **表单少即是多**：长表单分步，输入类型匹配键盘，错误提示贴近字段，不让用户猜错哪了

### 交互
- **触控目标不小于 44px**：图标按钮、列表项、输入控件都要能稳定点中
- **不依赖 hover**：提示、菜单、更多操作必须有点击或长按等移动端可触发方式
- **返回路径明确**：浏览器返回、应用内返回、弹窗关闭、流程取消要一致，避免误退丢数据
- **弱网可用**：提交按钮防重复、请求超时可重试、关键结果要有明确成功或失败反馈
- **分享与打开场景明确**：如果依赖 WebView 能力，要设计普通浏览器下的降级体验

### 视觉
- **字号保证可读**：正文通常不低于 14px，主要内容优先 16px；说明文字不要小到像免责声明
- **底部固定元素克制**：底栏、客服、悬浮 CTA 不要层层叠加遮住内容
- **图片轻量化**：优先 WebP/AVIF，首屏图设定尺寸和懒加载策略，避免布局跳动
- **活动页也要真实**：营销 H5 可以更有表现力，但按钮、表单、规则说明仍需清楚可靠

## 电脑版网页

适用于桌面浏览器中的 SaaS、后台、BI、CRM、控制台、内容管理、开发者工具和办公系统。

### 布局
- **信息架构先行**：主导航、二级导航、面包屑、详情页、编辑页、设置页的层级要稳定
- **宽屏有效利用**：常用工作区适配 1280px、1440px、1920px；不要把所有内容挤成窄卡片
- **密度服务任务**：运营、后台、数据产品应更紧凑，营销和内容页可以更舒展
- **表格是工作台**：列配置、筛选、排序、搜索、分页、批量操作、空状态和错误态要完整
- **详情与列表联动**：高频查看场景优先考虑抽屉、分栏或快捷预览，减少来回跳转

### 交互
- **键盘可用**：搜索、保存、关闭、翻页、选择、命令面板等高频操作应支持快捷键或焦点流
- **鼠标状态完整**：hover、active、disabled、focus、selected、loading 等状态不能缺
- **危险操作分级**：删除、覆盖、发布、退款、停用等操作要有确认、权限和可恢复策略
- **长任务可观测**：导入、导出、批处理、生成任务要展示进度、结果和失败详情
- **复杂筛选可复现**：筛选条件、视图、排序和列设置应能保存或通过 URL 分享

### 视觉
- **避免营销化后台**：内部工具少用大 hero、大装饰卡片和浮夸渐变，优先可扫描、可比较、可重复操作
- **颜色表达有边界**：状态色、品牌色、警示色不要混用；红色只给真正危险或失败的状态
- **图表服务问题**：图表必须有单位、时间范围、空值说明和异常值表达，不做只好看的统计图
- **响应式不断裂**：窄屏至少保证主流程可用，必要时隐藏次要列或改为卡片列表

## macOS App

适用于 SwiftUI/AppKit 原生 macOS 应用、菜单栏工具、桌面生产力工具和跨端桌面客户端。

### 平台习惯
- **遵循 macOS Human Interface Guidelines**：窗口、菜单栏、工具栏、侧边栏、设置页和系统控件尽量贴近平台习惯
- **菜单栏不是摆设**：常用命令、偏好设置、帮助、窗口管理、导入导出等应出现在菜单栏
- **快捷键要成体系**：保存、打开、关闭、搜索、撤销、重做、偏好设置等遵循 `Command` 系约定
- **窗口可调整**：设计最小尺寸、默认尺寸、全屏、分栏伸缩和多窗口/单窗口策略
- **系统权限透明**：文件、通知、网络、麦克风、辅助功能等权限要解释用途，并提供失败后的恢复路径

### 布局
- **侧边栏承载结构**：文档型、邮箱型、项目型工具优先使用侧边栏 + 内容区 + 检查器的经典结构
- **工具栏放高频命令**：图标按钮要有 tooltip，危险命令不要和常用命令挤在一起
- **设置页分组清晰**：偏好设置使用平台熟悉的分组和控件，避免把网页表单原样搬进来
- **状态栏轻量反馈**：同步状态、版本、连接状态、选中数量等可以放在底部或工具栏辅助区域
- **空状态可行动**：首次启动、无文档、无权限、无网络时给出下一步操作，而不是一片白地

### 视觉
- **原生控件优先**：按钮、列表、表格、菜单、分段控件、弹窗尽量使用系统组件
- **克制使用毛玻璃**：vibrancy/透明效果只用于符合系统语义的位置，不拿它糊满全屏
- **深浅色都要验证**：颜色、阴影、分隔线、图标和代码高亮需要适配系统外观
- **图标贴合平台**：App icon、toolbar icon、menu icon 尺寸和风格遵循 macOS 规范

## Android App

适用于原生 Android、Kotlin/Jetpack Compose、混合 App 的 Android 端和移动端主体验。

### 平台习惯
- **遵循 Material Design 3**：颜色系统、排版、组件、动效和状态反馈优先使用 Material 规范
- **系统返回必须正确**：返回键、手势返回、顶部返回、弹窗关闭和未保存拦截要一致
- **权限渐进请求**：通知、定位、相机、存储、通讯录等权限只在需要时请求，并解释收益
- **适配系统栏**：状态栏、导航栏、沉浸式、键盘弹起和手势区域要逐页验证
- **支持深色模式**：颜色 token、图片、图标和阴影都要在浅色/深色下可读

### 布局
- **触控目标不小于 48dp**：按钮、列表项、复选框、图标按钮都要满足基本可点面积
- **导航层级明确**：底部导航承载 3-5 个一级入口，复杂流程用顶部栏、抽屉或导航图管理
- **列表性能优先**：长列表使用分页、骨架屏、占位尺寸和增量加载，避免滚动卡顿
- **大屏与横屏不崩**：平板、折叠屏和横屏至少保证主流程可用，必要时使用双栏布局
- **离线和弱网可恢复**：本地缓存、重试、草稿保存、同步冲突提示要按业务风险设计

### 视觉
- **动态颜色谨慎用**：支持 Material You 时要保证品牌识别和状态色语义不被系统主题冲掉
- **动效短而明确**：页面转场、加载、展开收起要帮助理解层级，不做让人等的表演
- **通知可管理**：通知分渠道、分优先级，文案清楚，点击后落到正确页面
- **错误提示可行动**：Snackbar、Dialog、表单错误和空状态都要告诉用户下一步能做什么

## 多端一致性
- **一致的是任务，不是像素**：多端保持同一业务语义、信息结构和状态模型，但允许按平台重排
- **共享设计 token**：颜色、字号、间距、圆角、阴影、状态色、图标语义应尽量 token 化
- **组件按平台落地**：同一组件在 Web、macOS、Android 上可以有不同控件形态，但行为和文案要一致
- **文案统一管理**：多端文案统一术语表，避免一个功能在不同端叫三个名字
- **埋点口径统一**：关键流程的事件名、属性、成功/失败定义应跨端一致
- **验收按设备矩阵**：每次 UI 变更至少覆盖主流尺寸、深浅色、空/满数据、错误态和权限态
