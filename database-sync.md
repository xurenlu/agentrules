# 数据库同步最佳实践

## 同步策略选择
- **全量同步**：小数据量，定时覆盖。简单可靠，适合 < 10 万条且变化不大的数据
- **增量同步**：只同步变更部分。用时间戳、版本号、binlog/WAL 来追踪
- **实时同步**：CDC（Change Data Capture）方案。Debezium + Kafka、Canal、 Maxwell
- **双向同步**：能不来就别来。冲突处理复杂度指数增长，绝大多数场景不需要

## 增量同步设计
- **必须有可靠的变更标记**：
  - `updated_at` 时间戳：简单但有精度问题（同一秒多条）
  - 自增版本号：精确但需要应用层维护
  - binlog 位点：最可靠，但依赖数据库类型
- **处理删除**：软删除（`deleted_at`）+ 定期清理。硬删除要靠 binlog 捕获
- **断点续传**：记录最后一次同步的 checkpoint，挂了能继续
- **顺序很重要**：INSERT 后再 UPDATE 和 UPDATE 后再 INSERT 结果不同。建议按操作时间序

## 数据一致性
- **校验和对比**：同步完做 checksum。pt-table-checksum（MySQL）或者自己算 MD5
- **最终一致性 > 强一致性**：跨系统同步别追求实时强一致，不现实。定义可接受的延迟窗口
- **幂等是关键**：每条记录同步操作可以安全重放。用唯一键 + ON DUPLICATE KEY UPDATE 或 upsert
- **补偿机制**：定时任务扫描不一致数据并修复。别指望同步永远不出错

## 冲突解决（双向同步）
- **Last Write Wins (LWW)**：简单粗暴，后更新的覆盖前面的。适合数据冲突影响小的场景
- **CRDT**：无冲突数据类型。复杂但理论上完美。实际落地用成熟的库
- **应用层合并**：业务逻辑决定怎么合。比如"金额只增不减"、"状态只能往前流转"
- **人工介入**：冲突记录写入 conflict_log 表，通知人工处理。最笨但最靠谱

## 性能与可靠性
- **批量 > 逐条**：1000 条一批写入，减少网络往返和事务开销
- **限流**：同步不要打满目标库。控制 QPS 给正常业务留空间
- **超时与重试**：网络超时指数退避重试（最多 3 次），卡住的同步任务要有死信队列
- **监控告警**：
  - 同步延迟：超过阈值告警（如延迟 > 5 分钟）
  - 同步失败率：连续失败 N 条告警
  - 数据差异：定时 checksum 不一致告警

## 工具矩阵
| 场景 | 推荐方案 |
|------|---------|
| MySQL → MySQL 实时 | Canal + Kafka |
| PostgreSQL → 数据仓库 | Debezium + Kafka Connect |
| MySQL → Elasticsearch | go-mysql-elasticsearch / Logstash |
| 跨云数据同步 | DTS（阿里云 DTS / AWS DMS） |
| 应用层定时同步 | 自研定时任务 + 批量 upsert |
| 跨系统 ETL | Apache NiFi / Airbyte / 自研脚本 |

## 上线检查清单
- [ ] 同步方向明确（单向/双向/多向）
- [ ] 冲突处理策略已定义且测试通过
- [ ] 初始全量同步已完成并校验
- [ ] checkpoint/断点续传机制可用
- [ ] 监控告警已配置
- [ ] 回滚方案已准备（关闭同步不影响业务）
- [ ] 目标库容量评估（不会因为同步撑爆磁盘）
