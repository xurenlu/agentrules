# AI 协作红线

面向 AI 编程助手的行为红线。这些规则与具体语言、框架无关，约束 AI 在任何仓库里的操作行为；所有项目画像默认包含本文件。

## 硬约束

> 本节是 AI 必守清单，违反即错误；下文各节是背景与细节，按需阅读。

- **MUST**：创建/更新文件用编辑/写文件能力（直接写入文件、应用补丁），确保变更可审计、可回滚、可 review
- **MUST NOT**：用管道、`cat`、`echo` 重定向、heredoc 等 shell 方式生成或覆写文件内容
- **MUST**：破坏性或不可逆操作先向用户确认后再执行：删除数据、覆盖未读过的文件、强制推送、对外发布、执行 `migrate down`
- **MUST NOT**：`git push --force` 到 main / develop 等共享分支
- **MUST NOT**：执行不带 WHERE 的 UPDATE / DELETE，或 DROP / TRUNCATE 等语句，除非用户明确要求且已确认备份可恢复
- **MUST NOT**：`rm -rf` 删除仓库外目录或用户数据；批量清理前先列出将删除的内容让用户确认
- **MUST**：如实报告执行结果——测试失败就说失败并附输出，跳过的步骤明确说明；不把未验证的工作说成已完成
- **MUST**：发现疑似密钥/敏感信息时优先脱敏或移除并提示风险，不复述到回复、日志或提交信息里
- **MUST NOT**：做与任务无关的改动（顺手重构、重排格式、改无关命名）；改动范围与任务对齐

## 为什么禁止 shell 方式写文件

`cat > file <<EOF`、`echo ... > file` 这类写法有三个问题：

- **不可审计**：变更混在命令历史里，diff 工具看不到写入前后的对比
- **易出错**：转义、变量插值、多行内容在 heredoc 里踩坑概率高，且失败往往是静默的
- **不可恢复**：重定向覆写没有中间态，写坏了就是坏了

用编辑/写文件能力（Write / Edit / apply patch）走正规文件变更通道，出问题可以回滚，review 时能看到完整 diff。

## 破坏性操作的判断标准

满足任一条即属于破坏性操作，执行前必须确认：

- 操作后无法通过简单手段恢复原状（删数据、覆盖文件、强推改写历史）
- 影响范围超出当前仓库（发布、部署、改共享配置、动生产数据库）
- 用户没有明确要求做这件事，只是"顺理成章推断出来"的

拿不准就问。多问一句的成本远低于恢复数据的成本。

## 汇报规则

- 结果导向：先说做成了什么/没做成什么，再说过程
- 失败不粉饰：报错原文、失败的测试名、跳过的步骤都要如实给出
- 没验证的不承诺：没跑过的代码不说"已完成"，说"已实现，待验证"
