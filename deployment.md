# 服务部署最佳实践

## 核心原则
- **可回滚优先**：每次部署都必须能快速回到上一个稳定版本。发布不是摆谱，是留后路
- **小步发布**：能灰度就灰度，能分批就分批，别一脚油门把所有实例同时换掉
- **先验证再切流量**：新版本启动后必须通过健康检查、版本检查和关键接口探测，再对外承载流量
- **部署过程可观测**：部署日志必须记录版本号、构建产物、目标机器、开始/结束时间、健康检查结果和操作者/流水线信息
- **禁止静默失败**：部署、迁移、重启、健康检查任一步失败，都必须明确返回失败原因和下一步处理建议

## 发布前检查
- **确认版本号**：发布版本必须和前端、后端、API header、构建产物中的版本号一致
- **确认变更类型**：区分 feature、bugfix、hotfix、配置变更和数据库变更，别把几种大杂烩塞进一次发布
- **确认依赖和配置**：新版本需要的环境变量、外部服务、数据库权限、证书和系统依赖必须提前准备好
- **确认备份可用**：涉及数据库、文件存储或关键配置变更时，发布前必须确认最近备份可恢复
- **确认回滚路径**：发布前必须知道回滚到哪个版本、回滚命令是什么、是否需要回滚数据库迁移

## 数据库迁移
- **启动自动 up 一次**：数据相关 migration 一般在服务启动时自动执行一次 `migration up`
- **迁移失败即停止启动**：migration up 失败时服务不得继续提供流量，必须输出失败文件、迁移版本和错误上下文
- **提供命令行子命令**：服务或运维工具应提供 `migrate up`、`migrate down`、`migrate status`，并支持按步数回滚
- **多实例必须加锁**：多实例启动自动迁移时，必须使用数据库锁、迁移工具自带锁或分布式锁，确保只有一个实例执行
- **高风险迁移要审批**：大表 DDL、不可逆数据变更、删字段/删表必须人工审批，不能靠启动自动迁移一路莽过去

## 构建产物与备份
- **产物要可追溯**：构建产物必须能追溯到 git commit、版本号、构建时间和构建环境
- **替换前备份旧产物**：安装新版本前必须备份当前运行产物，二进制部署时备份旧二进制到带版本号/时间戳的目录
- **保留最近版本**：至少保留最近 3 个可回滚版本；磁盘紧张也别只留"信仰"
- **配置和程序分离**：配置文件、密钥、证书不得打进公开构建产物；敏感配置走环境变量、密钥管理或受控配置文件
- **部署脚本幂等**：同一部署命令重复执行，结果应该一致，不应制造重复备份、重复任务或脏状态

## 发布流程
- **停止或摘流量**：更新实例前先从负载均衡摘流量，或进入维护/只读/灰度状态
- **安装新版本**：备份旧版本后替换新产物，保持文件权限、属主和运行目录一致
- **启动服务**：启动时自动执行必要的 migration up，并记录迁移结果
- **健康检查**：检查进程存活、端口监听、健康接口、版本接口、关键业务接口
- **恢复流量**：健康检查通过后再恢复流量；失败则不要切流量，直接按回滚预案处理

## 部署后反馈
每次部署完成后，必须反馈以下信息：

- **发布版本**：新版本号、git commit、构建时间
- **部署结果**：成功/失败、目标机器、耗时、健康检查结果
- **迁移结果**：迁移前版本、迁移后版本、执行耗时、是否有失败或跳过
- **备份位置**：旧二进制/旧产物备份路径
- **回滚命令**：恢复旧产物、重启服务、检查健康状态的具体命令
- **数据库回滚判断**：是否需要执行 `migrate down`；如果需要，给出命令、步数和风险提示

## 回滚规则
- **先回滚应用，再考虑数据库**：大多数问题优先恢复旧二进制/旧产物并重启，不要一上来就动数据库
- **数据库回滚必须谨慎**：只有新迁移导致旧版本无法运行时，才在确认备份和影响范围后执行 `migrate down`
- **回滚也要健康检查**：回滚完成后必须验证版本、进程、日志、健康接口和关键业务接口
- **回滚过程要记录**：记录回滚原因、回滚版本、执行人、开始/结束时间、是否执行数据库 down
- **回滚后再复盘**：先恢复服务，再分析根因。线上事故不是写论文，先救火再讲究文采

## CI/CD 集成
- **流水线要分阶段**：构建、测试、打包、备份、发布、健康检查、回滚提示分阶段执行
- **失败自动中止**：测试、构建、迁移、健康检查失败时，流水线必须中止并保留日志
- **发布记录持久化**：部署记录应写入 CI/CD 日志、发布系统或变更记录，不能只靠聊天窗口里一句“好了”
- **生产发布要有审批**：高风险变更、数据库破坏性变更、跨服务联动发布必须人工确认
- **通知要有信息量**：发布通知必须包含版本、环境、结果、健康检查、备份路径和回滚方法

## 常用部署形态
| 场景 | 推荐做法 |
|------|---------|
| 单文件 Go 服务 | 备份旧二进制，替换新二进制，systemd 重启，健康检查 |
| Go + React embed | 前端构建产物 embed 进 Go 二进制，按单文件服务部署 |
| Docker 服务 | 使用不可变镜像 tag，保留旧镜像，滚动更新，健康检查失败自动回退 |
| K8s 服务 | Deployment 滚动发布，readiness/liveness probes，必要时 `kubectl rollout undo` |
| 静态前端 | 构建产物带版本目录，切换软链或对象存储版本，保留旧版本 |
